# Live Collaboration: P2P Binary Asset Exchange

## Problem

When users collaborate in a live room, the Yjs document syncs the box graph (metadata, structure, parameters) but **not** the binary assets (audio samples, soundfonts). Each user's assets live in their local OPFS. A new joiner sees `AudioFileBox` references with UUIDs pointing to files they don't have.

Assets must never be stored on the openDAW server to avoid hosting potentially infringing material. WebRTC data channels solve this: peers exchange binary data directly.

---

## Current State

**Loader pipeline** (both samples and soundfonts follow this):
```
Loader created for UUID
  → check OPFS (SampleStorage / SoundfontStorage)
  → miss → call provider.fetch(uuid, progress)
  → provider fetches from cloud (OpenSampleAPI / OpenSoundfontAPI)
  → loader saves result to OPFS
  → loader transitions to "loaded" state
```

**Signaling server** exists at `wss://live.opendaw.studio/signaling` — a generic topic-based pub/sub relay. It forwards JSON messages between subscribers. It does not inspect or store the message content. **No server changes needed.**

**Client-side WebRTC**: not implemented. Entirely greenfield.

**Yjs awareness**: already tracks connected clients per room. Provides peer count without server changes.

---

## Asset Types to Exchange

| Box Type | Provider Interface | Return Type | Zip Contents |
|----------|-------------------|-------------|-------------|
| `AudioFileBox` | `SampleProvider` | `Promise<[AudioData, SampleMetaData]>` | `audio.wav` + `meta.json` (peaks excluded — regenerated by manager) |
| `SoundfontFileBox` | `SoundfontProvider` | `Promise<[ArrayBuffer, SoundfontMetaData]>` | `soundfont.sf2` + `meta.json` |

---

## Constraint: YSync is a Public SDK Surface

`YSync` is used by SDK consumers. It must not break backwards compatibility. The P2P asset exchange is an additive layer — no modifications to `YSync.ts`, `YMapper.ts`, or the Yjs document structure.

## New Package: `packages/studio/p2p`

All P2P asset exchange code lives here. No existing packages are modified except `boot.ts` (wiring) and `StudioMenu.ts` (room join trigger).

## Exact Integration: How the Peer Provider Plugs In

### The Problem

Both managers freeze their provider at construction:

```typescript
// GlobalSampleLoaderManager.ts:29,35-36
readonly #provider: SampleProvider
constructor(provider: SampleProvider) {
    this.#provider = provider
}

// GlobalSoundfontLoaderManager.ts:7,10-11
readonly #provider: SoundfontProvider
constructor(provider: SoundfontProvider) {
    this.#provider = provider
}
```

They are created once in `boot.ts:76-83` and passed to `StudioService` as `readonly` fields. There is no setter.

### The Solution: `ChainedSampleProvider` / `ChainedSoundfontProvider`

New classes in `packages/studio/p2p` that implement the existing provider interfaces and are passed at boot time. They hold a mutable peer fallback slot.

```typescript
// packages/studio/p2p/src/ChainedSampleProvider.ts
import {SampleProvider} from "@opendaw/studio-core"
import {AudioData} from "@opendaw/lib-dsp"
import {Optional, Progress, UUID} from "@opendaw/lib-std"
import {SampleMetaData} from "@opendaw/studio-adapters"

export class ChainedSampleProvider implements SampleProvider {
    readonly #cloud: SampleProvider
    #peer: Optional<SampleProvider> = undefined

    constructor(cloud: SampleProvider) {
        this.#cloud = cloud
    }

    attachPeer(provider: SampleProvider): void {this.#peer = provider}
    detachPeer(): void {this.#peer = undefined}

    async fetch(uuid: UUID.Bytes, progress: Progress.Handler): Promise<[AudioData, SampleMetaData]> {
        try {
            return await this.#cloud.fetch(uuid, progress)
        } catch (cloudError: unknown) {
            if (isDefined(this.#peer)) {
                return this.#peer.fetch(uuid, progress)
            }
            throw cloudError
        }
    }
}
```

Same pattern for `ChainedSoundfontProvider` with `[ArrayBuffer, SoundfontMetaData]`.

### Wiring in `boot.ts`

```typescript
// boot.ts (modified lines 76-83)
const chainedSampleProvider = new ChainedSampleProvider({
    fetch: async (uuid: UUID.Bytes, progress: Progress.Handler): Promise<[AudioData, SampleMetaData]> =>
        OpenSampleAPI.get().load(uuid, progress)
})
const chainedSoundfontProvider = new ChainedSoundfontProvider({
    fetch: async (uuid: UUID.Bytes, progress: Progress.Handler): Promise<[ArrayBuffer, SoundfontMetaData]> =>
        OpenSoundfontAPI.get().load(uuid, progress)
})
const sampleManager = new GlobalSampleLoaderManager(chainedSampleProvider)
const soundfontManager = new GlobalSoundfontLoaderManager(chainedSoundfontProvider)
```

The `ChainedSampleProvider` and `ChainedSoundfontProvider` references must be accessible from the room join code. Two options:

**Option A**: Store them on `StudioService` as additional fields:
```typescript
// StudioService constructor gains two more parameters:
constructor(
    ...,
    readonly chainedSampleProvider: ChainedSampleProvider,
    readonly chainedSoundfontProvider: ChainedSoundfontProvider,
    ...
)
```

**Option B**: Create a `P2PSession` factory that takes the chained providers and is stored on `StudioService`:
```typescript
readonly p2p: P2PSessionFactory  // holds references to chained providers
```

Either way, `StudioMenu.ts` (which has access to `service: StudioService`) can call `attachPeer()` / `detachPeer()` when joining/leaving a room.

### Room Join Trigger

In `StudioMenu.ts:117-136`, after `YService.getOrCreateRoom()` succeeds:

```typescript
// After project is set:
const peerSession = new P2PSession(roomName, chainedSampleProvider, chainedSoundfontProvider)
await peerSession.connect()  // connects to /signaling, subscribes to topic
// peerSession calls chainedSampleProvider.attachPeer(peerAssetProvider)
// peerSession calls chainedSoundfontProvider.attachPeer(peerAssetProvider)

// On room leave / project close:
peerSession.disconnect()
// calls chainedSampleProvider.detachPeer()
// calls chainedSoundfontProvider.detachPeer()
```

### Full Data Flow: New Joiner Loads a Missing Sample

```
1. Yjs syncs box graph → AudioFileBox with UUID "abc" exists
2. Project creates loader → sampleManager.getOrCreate("abc")
3. GlobalSampleLoaderManager.#load() → SampleStorage.load("abc") → OPFS miss
4. GlobalSampleLoaderManager.#fetchFromApi() → this.#provider.fetch("abc", progress)
5. #provider is ChainedSampleProvider:
   a. Try cloud: OpenSampleAPI.load("abc") → 404 (user-uploaded asset, not in cloud)
   b. Cloud throws → check #peer → PeerSampleProvider is attached
   c. Call PeerSampleProvider.fetch("abc", progress)
6. PeerSampleProvider:
   a. Broadcasts via signaling: { type: "asset-request", missing: ["abc"] }
   b. Peers respond: { type: "asset-inventory", peerId: "xyz", have: ["abc"] }
   c. Establishes RTCPeerConnection with peer "xyz" via signaling (SDP/ICE exchange)
   d. Peer "xyz" reads audio.wav + meta.json from its OPFS, zips them, chunks the zip
   e. Chunks arrive over data channel, progress reported via fetchProgress callback
   f. All chunks received → reassemble zip → unzip → decode WAV to AudioData, parse meta.json
   g. Return [AudioData, SampleMetaData]
7. Back in GlobalSampleLoaderManager.#fetchFromApi():
   a. Receives [audio, meta] from provider
   b. Generates peaks via Workers.Peak.generateAsync()
   c. Saves to SampleStorage (OPFS)
   d. Caches in memory, calls loader.setLoaded()
8. UI updates: waveform appears
```

## Signaling Protocol

All messages go through the existing `/signaling` endpoint as `publish` payloads. The server relays them opaquely. Clients subscribe to topic `"assets:{roomName}"`.

### Message Types (client ↔ client, via signaling relay)

```typescript
// Discovery
type AssetRequest = {
    type: "asset-request"
    peerId: string                    // sender's ID (from doc.clientID)
    missing: Array<string>            // UUID hex strings
}

type AssetInventory = {
    type: "asset-inventory"
    peerId: string
    have: Array<string>               // UUIDs this peer can provide
}

type AssetAvailable = {
    type: "asset-available"
    peerId: string
    uuid: string                      // newly imported asset
}

// WebRTC connection establishment
type RTCOffer = {
    type: "rtc-offer"
    peerId: string
    targetPeerId: string
    sdp: string
}

type RTCAnswer = {
    type: "rtc-answer"
    peerId: string
    targetPeerId: string
    sdp: string
}

type RTCIceCandidate = {
    type: "rtc-ice-candidate"
    peerId: string
    targetPeerId: string
    candidate: RTCIceCandidateInit
}

// Transfer coordination
type AssetResume = {
    type: "asset-resume"
    uuid: string
    fromChunk: number
}
```

### Peer Count Discovery

Use Yjs awareness (already available on the `WebsocketProvider`) to know how many peers are in the room. No server changes needed. When broadcasting `asset-request`, wait for responses until:
- All awareness-known peers have responded, OR
- 3 second timeout (for slow peers)

## Data Channel Binary Protocol

Once an RTCPeerConnection is established, asset data flows over a data channel named `"assets"` configured as `ordered: false, maxRetransmits: 3`.

### Message Format

```
┌──────────┬──────────┬──────────┬──────────────────┐
│ msg type │ asset ID  │ chunk #  │ payload           │
│ 1 byte   │ 16 bytes │ 4 bytes  │ up to 64KB        │
└──────────┴──────────┴──────────┴──────────────────┘

msg type:
  0x01 = chunk-data
  0x02 = transfer-start  (payload = JSON: { totalChunks, zipSize, fileName })
  0x03 = transfer-complete
  0x04 = chunk-ack
  0x05 = cancel
```

### Chunk Size

64 KB (well under WebRTC's 256 KB SCTP limit).

### Backpressure

Monitor `dataChannel.bufferedAmount`. Pause sending above 1 MB, resume below 256 KB.

### Zip Contents

Sender reads asset files from OPFS, creates a zip using JSZip (already a dependency via `ProjectBundle`):
- **Sample**: `audio.wav` + `meta.json`
- **Soundfont**: `soundfont.sf2` + `meta.json`

Receiver unzips, then:
- **Sample**: decode WAV → `AudioData`, parse `meta.json` → `SampleMetaData`, return `[AudioData, SampleMetaData]`
- **Soundfont**: raw `ArrayBuffer` from sf2 + parse `meta.json` → `SoundfontMetaData`, return `[ArrayBuffer, SoundfontMetaData]`

### Progress Reporting

As chunks arrive, `PeerSampleProvider` / `PeerSoundfontProvider` calls the `progress: Progress.Handler` callback with `receivedChunks / totalChunks`. This flows into `SampleLoaderState { type: "progress", progress: unitValue }` / `SoundfontLoaderState`, which the UI already renders.

## Swarm Transfer Strategy

### Asset Registry (per room, in-memory on each client)

```typescript
type AssetEntry = {
    uuid: UUID.Bytes
    providers: Set<string>              // peer IDs that have this asset
    assignedTo: Optional<string>        // peer currently sending to us
    receivedChunks: Set<number>
    totalChunks: number
    buffers: Map<number, ArrayBuffer>   // chunk index → data
}

type AssetRegistry = Map<string, AssetEntry>  // keyed by UUID hex string
```

### Assignment

When a joiner needs N assets from M peers:
1. Round-robin by estimated byte totals per peer.
2. Prefer peers with more assets (fewer connections needed).
3. One active transfer per peer at a time.
4. Queue remaining; each peer works through its queue.

### Failure Handling

| Failure | Detection | Recovery |
|---------|-----------|----------|
| Peer disconnects mid-transfer | `oniceconnectionstatechange` → `"disconnected"` / `"failed"` | Reassign to another provider. Resume from last acknowledged chunk. |
| Peer disconnects before starting | Signaling `close` event | Remove from registry. Reassign queued assets. |
| Network stall (10s no chunks) | Timer on data channel | Mark stalled. Reassign to next provider. Keep connection alive. |
| All providers gone | Registry check after departure | Set loader to error state: `{ type: "error", reason: "No peers available" }` |
| New peer joins mid-transfer | New `asset-inventory` message | Add to registry. Rebalance queued (not in-progress) assets. |
| Integrity check fails | SHA-256 mismatch after reassembly | Discard. Re-request from different peer. After 2 failures, error. |

### Mid-Session Asset Import

When a user imports a new sample or soundfont during an active session, broadcast `asset-available` via signaling. Other peers add it to the registry so future loader requests can find it.

## STUN / TURN

```typescript
const rtcConfig: RTCConfiguration = {
    iceServers: [
        { urls: "stun:stun.l.google.com:19302" },
        { urls: "turn:live.opendaw.studio:3478", username: "opendaw", credential: "<secret>" }
    ]
}
```

- **STUN**: free, public Google server. Enables NAT traversal for direct P2P.
- **TURN**: coturn (open-source, BSD license) on the Hetzner VPS. Fallback relay for peers behind symmetric NATs (~8-15% of connections). Data transits through server memory but is never stored.

coturn setup: see [deploy-server.md](deploy-server.md).

## Alternatives Considered

### y-webrtc
Not suitable. Designed for syncing Yjs CRDT updates (small deltas), not large binary transfers.

### WebTorrent
Overkill for 2-10 peers. Heavy dependency, torrent semantics awkward for ephemeral sessions.

### PeerJS / simple-peer
We already have a signaling server. PeerJS adds an external dependency. simple-peer is unmaintained.

### WebTransport
Requires server relay (contradicts design constraint). No Safari support.

### Hybrid: small assets via Yjs, large via WebRTC
Yjs documents are in server memory. Even small assets accumulate and bloat RAM.

**Decision**: Native `RTCPeerConnection` with the existing signaling server. No new dependencies beyond the new package.

### Phase 3: Swarm
- `AssetSwarm`: asset registry, assignment algorithm, failover.
- Multi-peer parallel downloads.
- Stall detection (10s timeout) and takeover.

- SHA-256 integrity verification.

- Mid-session `asset-available` broadcasts.

- Backpressure (`bufferedAmount` monitoring).

### Phase 5: Tests
- Unit tests for all pure-logic modules (ChunkProtocol, AssetSwarm, ChainedProvider, AssetZip).
- Integration test with local RTCPeerConnection pair.
- Failure scenario tests.

## Resolved Decisions

1. **Asset size limits**: Show a warning for assets > 50 MB. No hard cap.
2. **Room persistence**: Rooms deleted on server when all users leave. Assets in OPFS persist. Missing assets re-transferred on rejoin.
3. **TURN relay**: Yes. coturn on Hetzner VPS. Transient relay only, no storage.
4. **Encryption**: DTLS (built into WebRTC) is sufficient.
5. **Server changes**: None. The existing `/signaling` endpoint relays all messages opaquely. Peer count comes from Yjs awareness.
6. **Peaks in zip**: No. Peaks are derived data, regenerated by `GlobalSampleLoaderManager` after provider returns `AudioData`.
