# Fix #920 — YSync TypeError on pointer flush

## Problem

Crash report `id=920` (build `1f3ec206`, prod, 2026-04-18):

```
TypeError: Cannot read properties of undefined (reading 'set')
  at YSync.ts:265 (pointer branch)
    field.set(String(fieldKeys[fieldKeys.length - 1]),
              newAddress.mapOr(address => address.toString(), null))
```

The error fires inside `#setupOpenDAW`'s flush loop while transferring accumulated box-graph updates into Yjs. The call stack is:

```
BoxEditing.modify
  BoxGraph.endTransaction
    transactionListeners.onEndTransaction
      YSync.onEndTransaction
        doc.transact( this.#updates.forEach(update => { ... }) )
          pointer branch -> field.set(...)   <-- throws
```

### Why `field` is `undefined`

In the pointer branch:

```typescript
const boxObject = asDefined(this.#boxes.get(key), "Could not find box") as Y.Map<unknown>
const {address: {fieldKeys}, newAddress} = update
let field = boxObject.get("fields") as Y.Map<unknown>
for (let i = 0; i < fieldKeys.length - 1; i++) {
    field = asDefined(field.get(String(fieldKeys[i])),
        `No field at '${fieldKeys[i]}'`) as Y.Map<unknown>
}
field.set(String(fieldKeys[fieldKeys.length - 1]), ...)
```

`asDefined` panics on null or undefined, so if the loop ran it cannot produce `undefined`. The only path that reaches `field.set` with `field === undefined` is:

- `fieldKeys.length === 1` (loop skipped), AND
- `boxObject.get("fields")` returned `undefined`

The Yjs `boxes` map still has an entry for this UUID, but that entry lacks the `"fields"` sub-map. The invariant that every box map is `{name, fields}` is broken.

### Trigger from the log

```
MenuItem.trigger: "Duplicate"
got 1 local updates from '[openDAW] updates'          <-- duplicate synced
start(RegionMoveModifier{0, 0})
approve(RegionMoveModifier{deltaIndex: -1, deltaPosition: 7680})
[ClipResolver.#createSolver] tasks:[{type:"delete", regionP:38400, regionC:46080}]
got 1 local updates from '[openDAW] updates'          <-- partial commit
[ErrorHandler] TypeError ...
```

The crash sits inside the modify-block that moves the duplicated region across tracks and, via `RegionOverlapResolver` (behaviour `clip`), cascades `region.box.delete()` on the overlapping region. The delete cascade emits many `pointer.defer()` + `unstage` updates which are then flushed into Yjs.

### Two cooperating root causes

1. **Flush never recovers from an error.** `#setupOpenDAW`:

   ```typescript
   this.#getDoc().transact(() => this.#updates.forEach(...), "[openDAW] updates")
   this.#updates.length = 0     // <-- skipped when transact throws
   ```

   When `transact` rethrows, the queue is not cleared. The next `onEndTransaction` then replays the already-applied updates against a Yjs state that has moved on, which is a reliable source of "missing field map" drift. The earlier `Could not add to selectableMap` rollback in the same session is exactly the kind of hiccup that primes this.

2. **Raw update stream around phantom boxes.** YSync listens via `subscribeToAllUpdatesImmediate`, which bypasses `editing.optimizeUpdates` (phantom filter for UUIDs that are created and deleted in the same transaction). When a phantom box appears in the queue, the interleaving of `new`, `pointer`, and `delete` updates can leave `this.#boxes.get(key)` referencing a box Y.Map whose `fields` child was never integrated, or was already replaced.

Together, a bad update at the tail of the queue blows up before the queue is cleared, and every subsequent transaction amplifies the inconsistency.

## Solution

### 1. Always drain `this.#updates`

Wrap the flush so the queue is cleared even when `transact` throws. On error, capture the offending updates so we can log them and tell the user their action failed.

```typescript
onEndTransaction: (rolledBack) => {
    if (this.#ignoreUpdates || rolledBack) {
        this.#updates.length = 0
        return
    }
    const pending = this.#updates.splice(0)
    const result = tryCatch(() =>
        this.#getDoc().transact(() => pending.forEach(update => { ... }),
                                "[openDAW] updates"))
    if (result.status === "failure") {
        console.error("[YSync] flush failed, dropping updates", {
            count: pending.length,
            updates: pending.map(String),
            error: result.error
        })
        throw result.error
    }
}
```

This does not fix the crash but it stops the crash from poisoning the rest of the session.

### 2. Drop phantom updates before flushing

Adopt the same filter `BoxEditing.optimizeUpdates` uses: compute the set of UUIDs with both `new` and `delete` in the batch and skip every update whose UUID is in that set. These updates are no-ops from the remote peer's perspective (the box never existed for them), so omitting them is safe and removes the ordering hazard.

```typescript
const pending = optimizeUpdates(this.#updates.splice(0))
```

`optimizeUpdates` is already exported-adjacent in `lib/box/src/editing.ts`; either re-export it or inline the same 10 lines in YSync.

### 3. Harden the pointer and primitive branches

Replace the implicit casts with explicit `asDefined` so a shape mismatch gives a readable error instead of a generic TypeError, and defend against an update targeting a field path that does not exist in the current schema.

```typescript
const boxObject = asDefined(this.#boxes.get(key), `Missing box '${key}'`) as Y.Map<unknown>
let field = asDefined(boxObject.get("fields"),
    `Box '${key}' missing 'fields'`) as Y.Map<unknown>
```

Also assert that every intermediate `field` is a `Y.Map` (via `isInstanceOf(field, Y.Map)`); if it is not, log the path and skip the write instead of throwing inside `transact`.

### 4. Optional: rebuild on drift

If #1–#3 detect a structural mismatch, re-serialise the box from `boxGraph` and reinstall it with `this.#boxes.set(key, YMapper.createBoxMap(box))` before applying the update. This is a targeted self-heal rather than a full resync.

## Implementation order

1. `optimizeUpdates` extraction (or copy) + apply it in the YSync flush.
2. Wrap `transact` in `tryCatch`, clear queue in a `finally`, rethrow.
3. Replace casts with `asDefined` in both `pointer` and `primitive` branches.
4. Add the `Y.Map` runtime guards in the traversal loop.
5. Optional self-heal on mismatch.

## Risks and notes

- `optimizeUpdates` currently lives in `lib/box/editing.ts` as a module-private helper. Exporting it is a minor API surface change; alternatively inline the ~15 lines into YSync.
- The `tryCatch` + rethrow path changes which error object bubbles up. Make sure `BoxEditing.modify`'s outer `tryCatch` still receives the original error so history state stays consistent.
- Self-heal (step 4) should stay behind a debug flag until we are confident it does not mask real schema bugs.

## Follow-up: silent-skip side effect

The hardening in step 3 replaces the pre-fix TypeError with `console.warn` + skip. That keeps the session alive but introduces a known failure mode we need to watch for before it becomes a new bug report.

### The side effect

When `#applyUpdate` detects drift (missing box, missing `"fields"` sub-map, or a non-`Y.Map` value on the traversal path), it logs and returns. The local `BoxGraph` has already applied the change; yjs has not. On reload (or for remote peers) the change is invisible, which is silent data loss from the user's point of view.

This is not new behaviour compared with the old code (the original TypeError left yjs partially updated and raised an error that bubbled past YSync), but the old behaviour was loud. The new one is quiet.

### When it will bite

We expect silent skips to become a user-visible bug under one or more of the following conditions. Each is a distinct thing to verify before we consider this closed.

1. **Repeatable drift in a specific editing flow.** If the same user action produces a skipped update every time (not an intermittent multi-peer race), then the skip masks a real schema bug. Candidates: region-delete cascades through shared resources, paste of a partial subgraph, undo/redo across a remote rebase.
2. **Remote peer rebase.** Two peers edit the same region concurrently; CRDT merge leaves a box map in a shape one peer does not expect on its next flush. Skip fires, local change never propagates, authors disagree on the project state.
3. **Reload after a skip.** User performs an edit, YSync logs a warn, user saves or reloads; yjs replays its state and the edit is gone. No error, no dialog, no indication that anything was lost.
4. **Automation that depends on the update reaching yjs.** Anything that reads from the yjs doc (server persistence, export workers, second browser tab acting as a viewer) disagrees with the editing tab.

### Investigation tasks

Run these in order. Each feeds into the decision about whether the silent-skip policy should be kept, escalated, or replaced.

1. **Count skips in the wild.**
   - Upgrade the `console.warn` in `#applyUpdate` and `#resolveFieldMap` to `console.error` (unconditional) or route them through the error-handler pipeline that produced report #920. Correlate with build telemetry.
   - If any real session logs a skip, capture the full `Update` (type, address, new-value/new-address) and the offending box's current yjs shape at skip time. Ship that in the report payload.
   - Acceptance: we know whether skips happen to real users at all, and at what rate.
2. **Reproduce the drift locally.**
   - Take each reported skip and try to reconstruct the graph/yjs state that produced it in a test. Start from the existing "mirrors the crash scenario" test and walk backwards: remove manual corruption (`leafMap.delete("fields")`) and look for a real sequence of graph operations that arrives at the same broken shape.
   - Candidate scenarios to brute-force first:
     - Two rapid duplicate + move + cascade-delete cycles inside one modify call.
     - Undo of a failed flush (history entry exists locally; yjs state was partially rolled back).
     - External peer update that was rolled back via `#rollbackTransaction` while we still held pending outgoing updates for the same UUIDs.
   - Acceptance: we have a unit test that exercises the drift without any manual corruption. If we cannot write one, the skip is guarding against a condition we do not yet understand.
3. **Instrument the rollback path.**
   - `#rollbackTransaction` is the most plausible source of shape-drift. Add a structural assertion after each rollback step that the restored value (when the change was `"delete"`) is a `Y.Map` with both `"name"` and `"fields"`. If the assertion fails, capture the oldValue snapshot.
   - Acceptance: confirm or disprove "rollback leaves boxes without their `fields` child" as the root cause of the field-map being gone.
4. **Decide on self-heal vs. escalate.**
   - If the skip is covering a transient CRDT race: add targeted self-heal — re-serialise the box via `this.#boxes.set(key, YMapper.createBoxMap(box))` before the skipped write, so the next flush is clean and yjs matches the local graph.
   - If the skip is covering a real data bug: stop skipping; throw and surface the error to the user, and fix the source of the drift.
   - Acceptance: we have evidence for one of the two branches and the code matches.
5. **Divergence detector (optional, longer term).**
   - Periodic background check: for every box UUID in `boxGraph`, confirm `#boxes.has(uuid)` and that `boxObject.get("fields")` is a `Y.Map`. Report mismatches through the same channel as step 1.
   - Acceptance: running the detector in a dev build for a week surfaces zero mismatches in normal use.

### Exit criteria for removing the silent skip

Either of the following:

- One of steps 1–3 finds a concrete root cause and it is fixed. The skip can then be replaced with an assertion that will never fire, because the drift state is now impossible.
- Step 4 picks the self-heal branch, the heal has run for a release cycle without surfacing inconsistencies, and we are willing to treat the skip as a defence-in-depth measure rather than a gap in our understanding.

Until one of those holds, every YSync flush warning should be treated as a potential user-facing bug, not an "everything is fine" signal.
